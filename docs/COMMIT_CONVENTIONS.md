# 커밋 컨벤션

이 레포의 커밋 메시지 규칙이다. **AI·사람 공통**으로 따른다.
대부분의 작업을 AI가 수행하므로, 커밋 메시지가 일관되고 간결해야 변경 이력을 빠르게 읽을 수 있다.

> 핵심 원칙: **"코드 리뷰에서 볼 수 있는 내용은 커밋 메시지에 쓰지 않는다."**
> 커밋은 *무엇을·왜* 바꿨는지를 요약하고, *어떻게* 는 코드/PR에 맡긴다.

---

## 형식

```text
<type>(<scope>): <제목>

- <변경 요약 1>
- <변경 요약 2>
```

- **제목(첫 줄)**: 50자 이내. `type` 필수, `scope` 선택. 무엇을 했는지 한 줄로.
- **본문(bullet)**: 주요 컴포넌트/기능만. **최대 5~7개**, 각 항목 한 줄. 사소한 변경만 있으면 본문 생략 가능.
- 제목 끝에 마침표를 찍지 않는다. 제목과 본문 사이에 빈 줄 1개.

## type

| type | 용도 |
|------|------|
| `feat` | 기능 추가 |
| `fix` | 버그 수정 |
| `refactor` | 동작 변화 없는 구조 개선 |
| `perf` | 성능 개선 |
| `docs` | 문서·주석 |
| `test` | 테스트 추가·수정 |
| `chore` | 설정·빌드·툴링·의존성 |
| `style` | 포매팅(동작 무관) |
| `ci` | CI/워크플로 |

## scope (이 Godot 템플릿 예시)

선택 사항이지만 있으면 이력 파악이 쉽다. 영역에 맞춰 쓴다.

`player`, `enemy`, `iso`, `world`, `tilemap`, `ui`, `autoload`, `input`,
`state-machine`, `mcp`, `skills`, `claude`, `test`, `docs`, `deps`, `ci`

## 작성 원칙

**포함할 것** — 변경의 *무엇/왜* 를 사람이 읽을 단위로.

**제외할 것** (글로벌 규칙):

- ❌ 기술 스택 세부 (예: "TileMapLayer 사용", "rem 단위")
- ❌ 파일명·경로 (예: `player.gd`, `scenes/main.tscn`)
- ❌ 개수·통계 (예: "시그널 4개", "테스트 3개")
- ❌ 구현 방법론 (예: "덕 타이핑으로 FSM 구성")
- ❌ 하위 세부 (코드 리뷰/디프에서 보이는 모든 것)

## AI attribution 금지 (중요)

- ❌ `Co-Authored-By: Claude ...` 또는 모든 AI/도구 attribution 문구를 **절대** 넣지 않는다.
- ✅ 커밋 메시지는 순수하게 변경사항만 반영한다.
- 이 규칙은 `commit-commands` 플러그인의 `/commit`을 쓸 때도 동일하게 적용된다.

## 예시

✅ **좋은 예시 (간결)**

```text
feat(player): 아이소 8방향 이동 구현

- 입력 방향에 따른 이동 처리
- WASD·화살표 입력맵 연동
- Y-sort 정렬 그룹에 배치
```

```text
fix(iso): 좌클릭 타일 좌표가 어긋나는 문제 수정
```

```text
docs: Godot 공식 컨벤션 가이드 추가
```

❌ **나쁜 예시 (과도한 상세 — 제외 대상 위반)**

```text
feat: player.gd에 CharacterBody2D 기반 이동 구현

- Input.get_vector로 4개 액션 받아 velocity 계산 (move_left/right/up/down)
- move_and_slide() 호출, speed export 180.0
- scenes/main.tscn의 World(y_sort_enabled) 아래 Player 인스턴스 배치
- 시그널 4개 추가, 테스트 3개 작성
```

## 커밋 단위

- 한 커밋은 **하나의 논리적 변경**만 담는다(기능·수정이 섞이면 분리).
- 커밋은 **사용자가 요청할 때** 한다. 기본 브랜치에 직접 커밋하지 말고 작업 브랜치를 쓴다.
